Teststufen und Agilität

Einführung in die Teststufen im agilen Kontext

Beim agilen Testen wird das sequenzielle Durchlaufen der einzelnen Teststufen aufgebrochen. Die einzelnen Stufen sind zwar notwendig, aber sie können flexibel und inkrementell ausgeführt werden. Das Modell von Gregory und Crispin unterteilt Tests entlang zweier Dimensionen: Fachlichkeit vs. Technik und Grundfunktionalität vs. Gesamtes Produkt.

Die Dimensionen der Tests

Die Unterscheidung erfolgt entlang der folgenden Dimensionen:

  • Fachliche Tests: Testen Software gegen Geschäftsanforderungen (z.B. korrekte Berechnung eines Versicherungstarifs).
  • Technische Tests: Testen die Interaktion der Softwarebausteine und deren Leistung (z.B. Reaktionszeit).
  • Grundfunktionalität: Tests des „Happy Path“ der Software, der die Grundfunktionalität überprüft.
  • Gesamtes Produkt: Tests, die nicht nur den „Happy Path“ abdecken, sondern auch nicht-funktionale Anforderungen und unübliche Nutzungspfade prüfen.

Quadranten des agilen Testens

Erster Quadrant: Technisch-Teamunterstützende Tests

Unit-Tests prüfen die Funktionstüchtigkeit kleiner Softwarekomponenten und bilden das Fundament des agilen Testens. Sie werden früh und häufig durchgeführt, meist automatisiert mit Werkzeugen wie JUnit. Diese Tests sind wichtig für die frühe Aussagekraft über die Lauffähigkeit der Software.

Zweiter Quadrant: Fachlich-Teamunterstützende Tests

Fachlich-teamunterstützende Tests überprüfen die Software auf der Grundlage fachlicher Beispiele oder „Storys“. Diese Tests finden meist nach der Integration mehrerer Komponenten statt und können über API, GUI oder Batch-Verarbeitung angestoßen werden.

Dritter Quadrant: Fachlich-Produkthinterfragende Tests

Diese Tests prüfen die Software auf ungewöhnliche Testfälle und nicht-funktionale Anforderungen. Sie werden von professionellen Anwendenden durchgeführt und helfen, neue Anforderungen frühzeitig zu identifizieren.

Vierter Quadrant: Technisch-Produkthinterfragende Tests

Technisch-produkthinterfragende Tests bewerten nicht-funktionale Anforderungen wie Last, Performance, Sicherheit und Zuverlässigkeit. Sie erfordern komplexe Werkzeuge und erfahrene Techniker:innen, die die Software auf anspruchsvollen Testumgebungen untersuchen.
